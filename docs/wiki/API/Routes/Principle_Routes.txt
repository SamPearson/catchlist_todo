====== Principle Routes ======

===== Overview =====

API endpoints for managing principles. Principles represent high-level values, aspirations, or "why" statements. They're meant to be few in number and represent core motivations that can be linked to actions.

The idea: define what kind of person you want to be, then connect tasks/projects/routines to those ideals without them getting lost in a sea of low-level tags.

Examples: "Be dependable", "Prioritize health", "Continuous learning", "Financial independence"

All endpoints require authentication via JWT Bearer token.


===== Principle Endpoints =====

==== List Principles ====

**GET /api/principles**

Returns all principles for the authenticated user.

**Response:** Array of principle objects (200)


==== Create Principle ====

**POST /api/principles**

Creates a new principle.

| Field       | Type   | Required | Description                                      |
|-------------|--------|----------|--------------------------------------------------|
| title       | string | yes      | Principle statement (max 100 chars)              |
| description | text   | no       | What this principle means to you; clarification of your personal definition |
| reason      | text   | no       | Why this principle matters to you; motivation and inspiration |
| color       | string | no       | Hex color code. Default "#ffd700" (gold)         |

**Response:** Created principle object (201)

**Errors:** 400 if title is missing or empty


==== Get Principle ====

**GET /api/principles/<principle_id>**

Returns a specific principle by ID.

**Response:** Principle object (200), or 404 if not found


==== Update Principle ====

**PUT /api/principles/<principle_id>**

Updates a principle's properties.

| Field       | Type   | Description                                       |
|-------------|--------|---------------------------------------------------|
| title       | string | Principle statement (cannot be empty if provided) |
| description | text   | What this principle means to you                  |
| reason      | text   | Why this principle matters to you                 |
| color       | string | Hex color code                                    |

**Response:** Updated principle object (200), or 404 if not found

**Errors:** 400 if title is set to empty


==== Delete Principle ====

**DELETE /api/principles/<principle_id>**

Permanently deletes a principle and removes it from all attached entities.

**Response:** 204 No Content, or 404 if not found


===== Association Endpoints =====

==== Attach Principle to Entity ====

**POST /api/principles/attach**

Attaches a principle to an entity.

| Field        | Type    | Required | Description                              |
|--------------|---------|----------|------------------------------------------|
| principle_id | integer | yes      | ID of the principle to attach            |
| target_type  | string  | yes      | Entity type (see principled entities)    |
| target_id    | integer | yes      | ID of the target entity                  |

**Business Logic:**
* Idempotent: attaching the same principle twice has no effect
* Entity must support principles (have PrincipledMixin)
* Valid target types: task, project, calendar, routine, session

**Response:** { "success": true } (200)

**Errors:**
* 400 if required fields missing
* 404 if principle or target entity not found
* 422 if target_type is invalid (returns { "error": "...", "code": "INVALID_TARGET_TYPE" })


==== Detach Principle from Entity ====

**POST /api/principles/detach**

Removes a principle from an entity.

| Field        | Type    | Required | Description                              |
|--------------|---------|----------|------------------------------------------|
| principle_id | integer | yes      | ID of the principle to detach            |
| target_type  | string  | yes      | Entity type (see principled entities)    |
| target_id    | integer | yes      | ID of the target entity                  |

**Business Logic:**
* Idempotent: detaching a non-attached principle has no effect
* Valid target types: task, project, calendar, routine, session

**Response:** { "success": true } (200)

**Errors:**
* 400 if required fields missing
* 404 if principle or target entity not found
* 422 if target_type is invalid (returns { "error": "...", "code": "INVALID_TARGET_TYPE" })


===== Principled Entities =====

The following entities can have principles attached:

| Entity Type | Description                        |
|-------------|------------------------------------|
| task        | A task (standalone or subtask)     |
| project     | A project                          |
| calendar    | A calendar container               |
| routine     | A routine template                 |
| session     | A specific routine session occurrence |


===== Principle Object =====

| Field       | Type     | Description                                  |
|-------------|----------|----------------------------------------------|
| id          | integer  | Primary key                                  |
| user_id     | integer  | Owner                                        |
| title       | string   | The principle statement (e.g., "Be dependable") |
| description | text     | What this principle means to you; clarification of your personal definition |
| reason      | text     | Why this principle matters to you; motivation and inspiration |
| color       | string   | Hex color code (default: #ffd700)            |
| created_at  | datetime | When created                                 |
| updated_at  | datetime | When last modified                           |


===== Design Philosophy =====

**Why Two Systems?**

A single tagging system could technically handle both use cases, but separating them serves important purposes:

1. **Prevents clutter**: Principles won't get buried among dozens of organizational tags
2. **Encourages intentionality**: Having a dedicated "principles" concept prompts reflection on values and motivations
3. **Enables different UI treatment**: Principles can be displayed prominently as motivational anchors; tags can remain more utilitarian
4. **Supports different queries**: "Show me everything tied to this principle" enables alignment checking; "filter by these tags" enables practical organization

**Intended Usage Pattern**

* **Tags**: Create freely as needed; use for filtering, organization, and categorization. Common, numerous, low-friction.
* **Principles**: Define thoughtfully; they represent aspirational values or ideals. Use to anchor work to what matters most. Fewer in number, higher in intent.

**Using Principles for Motivation**

Principles come with description and reason fields designed to be motivational touchstones:
* **Description**: Clarifies what the principle means in your own words
* **Reason**: Captures why this principle matters—motivation to draw on when planning or reviewing
* Together, they create a pocket motivation statement you can reference when work feels misaligned or uncertain
